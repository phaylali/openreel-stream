# Plan: Add RTMP Livestreaming Capability to OpenReel

## 1. Project Overview & Architecture

### Current State
OpenReel is a browser-based video editor using React, TypeScript, WebCodecs, and WebGPU. It has:
- Client-side rendering with canvas/WebGPU
- Recording via MediaRecorder API
- Export via WebCodecs/FFmpeg.wasm
- Secure storage service for API keys (AES-256-GCM)
- Cloudflare Pages Functions for stateless API proxying
- No existing streaming/RTMP infrastructure

### Goal
Add a "Go Live" button next to the Record button that streams the editor canvas to Twitch via RTMP. Additionally, add a Streaming configuration tab in Settings for Twitch credentials and ingest selection.

### Architecture Decision

**Browsers cannot directly stream via RTMP.** The solution requires a server-side component:

```
Browser (WebCodecs + WebSocket) → Streaming Server (Node.js + FFmpeg) → Twitch RTMP Ingest
```

**Streaming Pipeline:**
1. Capture canvas frames via WebGPU renderer or `canvas.captureStream()`
2. Encode using WebCodecs `VideoEncoder` (H.264/AVC)
3. Capture audio via Web Audio API (`AudioContext.createMediaStreamDestination()`)
4. Package video+audio into a container format (FLV) in a Web Worker
5. Send encoded chunks via WebSocket to local/remote streaming server
6. Server receives chunks, pipes to FFmpeg stdin
7. FFmpeg outputs to Twitch RTMP URL: `rtmp://<ingest>/app/<stream_key>`

**Why not MediaRecorder?** MediaRecorder outputs file fragments, not suitable for real-time streaming. WebCodecs provides frame-level control and hardware acceleration.

**Why WebSocket over WebRTC?** Simpler one-way streaming, no NAT traversal needed for local server, easier debugging. WebRTC adds complexity (SFU, STUN/TURN) unnecessary for this use case.

## 2. Implementation Phases

### Phase 1: Core Streaming Infrastructure (Client-Side)

#### 1.1 Create Streaming Service (`apps/web/src/services/streaming-service.ts`)
**Purpose:** Manage WebCodecs encoding and WebSocket transmission.

**Responsibilities:**
- Configure `VideoEncoder` with H.264 codec
- Capture canvas stream: `previewCanvas.captureStream(fps)` or directly from WebGPU renderer
- Capture audio: mix project audio tracks + mic (optional) via Web Audio API
- Encode frames and send via WebSocket
- Handle connection lifecycle (connect, disconnect, errors)
- Report streaming statistics (bitrate, dropped frames, latency)

**Key API:**
```typescript
class StreamingService {
  connect(serverUrl: string): Promise<void>
  start(streamConfig: StreamConfig): Promise<void>
  stop(): void
  pause(): void
  resume(): void>
  getStats(): StreamStats
  on(event: string, handler: (data) => void): () => void
}
```

**Dependencies:**
- `webcodecs-encoder` library (for easier WebCodecs usage) OR custom implementation
- WebSocket client (native `WebSocket`)
- Web Audio API for audio capture

**Notes:**
- Run encoding in a Dedicated Web Worker to avoid blocking UI
- Use `VideoFrame` from canvas via `new VideoFrame(canvas, { timestamp })`
- Audio: create `MediaStream` from Web Audio `MediaStreamAudioDestinationNode`

#### 1.2 Streaming Store (`apps/web/src/stores/streaming-store.ts`)
**Purpose:** Zustand store for streaming state management.

**State:**
- `isStreaming`: boolean
- `isPaused`: boolean
- `status`: "idle" | "connecting" | "live" | "paused" | "stopping" | "error"
- `error`: string | null
- `stats`: { duration, bitrate, framesSent, droppedFrames } | null
- `config`: current streaming configuration (resolution, fps, bitrate, audioEnabled)
- `connection`: WebSocket reference (non-persisted)

**Actions:**
- `startStream(config)`: initiate connection and encoding
- `stopStream()`: clean shutdown
- `pauseStream()` / `resumeStream()`
- `updateConfig(partialConfig)`
- `setError(message)`
- Event subscriptions for stats updates

**Persistence:** None (session-only). Similar to recorder-store.

#### 4.1 Server Directory Structure
Create `apps/streaming-server/` as a separate Node.js application in the monorepo:

```

    package.json
    tsconfig.json
    wrangler.toml (if deploying as Worker — but Workers unsuitable, so separate deployment)
    Dockerfile (optional)
    README.md
```

#### 4.2 Server Core Logic

**WebSocket Server (`src/websocket/server.ts`):**
- Uses `ws` library
- Listens on `PORT` (default 8080)
- On connection: creates `StreamSession` object
- Receives binary messages (encoded video/audio chunks)
- Forwards to `FFmpegProcess` instance
- Handles control messages (START, STOP, PAUSE) via JSON frames

**FFmpeg Process (`src/ffmpeg/ffmpeg-process.ts`):**
- Spawns `ffmpeg` child process
- Input: stdin (raw H.264 stream + AAC audio)
- Output: RTMP URL provided by client or from Twitch ingest
- Arguments:
```bash
ffmpeg
  -i -                          # stdin
  -c:v copy                     # passthrough (already H.264)
  -c:a aac                      # encode audio to AAC
  -f flv                        # FLV container for RTMP
  -flvflags no_duration_filesize # for streaming
  <rtmp_url>
```
- Monitors process exit, errors, and logs
- Restart logic on failure?

**RTMP Connection Manager (`src/ffmpeg/rtp-connection.ts`):**
- Validates RTMP URL (must be from Twitch ingests)
- Handles authentication (stream key in URL path)
- Reconnection logic (exponential backoff)
- Health reporting back to client via WebSocket

**Session Management:**
- Map `WebSocket` → `FFmpegProcess`
- One active stream per connection
- Cleanup on disconnect

#### 4.3 Security Considerations
- **Authentication:** Secret token in WebSocket handshake header (`Sec-WebSocket-Protocol` or query param) to prevent unauthorized streaming
- **Rate limiting:** Max connections per IP
- **URL validation:** Ensure RTMP URL matches known Twitch ingests pattern (or allow custom for future platforms)
- **Stream key handling:** Never log stream keys; sanitize error messages
- **TLS:** Recommend wss:// in production

#### 4.4 Deployment Options
**Option A: Standalone Node.js Server (Recommended)**
- Deploy on VPS (DigitalOcean, Linode, AWS EC2)
- Docker container for easy deployment
- Systemd service for process management
- User runs: `npm run start` in streaming-server directory

**Option B: Cloudflare Worker with WebSocket + External FFmpeg**
- Not feasible: Workers cannot spawn FFmpeg or maintain long-lived processes

**Option C: Serverless FFmpeg (e.g., AWS Lambda)**
- Complex cold-start, cost, duration limits

**Recommendation:** Provide a simple Docker-based server that users can run locally or on a VPS. Default to `localhost:8080` for local streaming.

#### 4.5 Server Configuration
Environment variables:
- `PORT=8080`
- `AUTH_TOKEN` (optional): shared secret for WebSocket auth
- `MAX_BITRATE` (optional): limit to prevent abuse
- `FFMPEG_PATH` (optional): custom ffmpeg binary path (default: `ffmpeg` in PATH)

### Phase 5: Integration & Polish

#### 5.1 Client-Server Connection Flow
1. User clicks "Go Live" → opens StreamingDialog
2. Dialog fetches settings from `settingsStore.streamingSettings`
3. User clicks "Go Live" → `streamingService.connect(wsUrl)` established
4. Service creates WebWorker for WebCodecs encoding
5. Worker:
   - Gets canvas stream: `canvas.captureStream(fps)`
   - Creates `VideoEncoder`, feeds frames from canvas
   - Creates `MediaStream` with audio tracks
   - Sends encoded chunks as binary WebSocket messages
   - Emits stats events back to main thread
6. Main thread updates `streamingStore.stats`
7. Server receives chunks, pipes to FFmpeg → RTMP

#### 5.2 Performance Optimizations
- **OffscreenCanvas:** Transfer canvas control to worker if possible: `canvas.transferControlToOffscreen()` — allows rendering and capturing in worker. However, current editor renders to visible canvas on main thread. For streaming, we can use visible canvas frames directly.
- **Frame skipping:** If encoder queue builds up, drop frames to maintain real-time
- **Adaptive bitrate:** (Future) Based on network conditions, adjust bitrate
- **Audio offloading:** Audio encoding in separate worker thread

#### 5.3 Error Handling & Recovery
- **Network disconnect:** Attempt reconnection with backoff; show user message
- **FFmpeg crash:** Server restarts process; notify client
- **Twitch reject:** RTMP handshake failure → display error
- **Browser incompatibility:** Check WebCodecs support at startup; show fallback message if unavailable

#### 5.4 User Experience
- Tooltip on disabled "Go Live" button: "Set up Twitch in Settings first"
- Toast notifications: "Stream started", "Stream ended", "Connection lost"
- Confirmation dialog before stopping stream (prevent accidental stop)

## 4. Technical Specifications

### 4.1 Video Encoding Settings
**Codec:** H.264 (AVC) Baseline profile, level 3.1 (compatible with Twitch)
**Resolution:** User-selectable; default 1080p (1920×1080)
**Frame Rate:** 30 or 60 fps (based on project timeline fps)
**Bitrate:** Based on resolution:
- 720p: 4,000 kbps
- 1080p: 6,000 kbps
- 1440p: 10,000 kbps
- 4K: 20,000 kbps
**Keyframe Interval:** 2 seconds (e.g., every 60 frames at 30fps)

### 4.2 Audio Encoding Settings
**Codec:** AAC-LC
**Sample Rate:** 48 kHz
**Channels:** Stereo (2)
**Bitrate:** 160 kbps (or user-selectable 96/128/192)

### 4.3 Container Format
**Intermediate:** FLV (Flash Video) via FFmpeg pipe
**Final Transport:** RTMP (Real-Time Messaging Protocol)

### 4.4 WebSocket Message Format
**Binary Messages:** Raw `EncodedVideoChunk` data + metadata via length-prefixing
**Control Messages (JSON):**
```json
{
  "type": "start" | "stop" | "pause" | "resume" | "config",
  "data": { ... }
}
```
**Stats Messages (JSON from server):**
```json
{
  "type": "stats",
  "data": {
    "bitrate": 6000000,
    "framesSent": 1234,
    "droppedFrames": 2,
    "timestamp": 1234567890
  }
}
```

## 5. Security & Privacy

### Twitch Stream Key Storage
- Stored in secure storage (same as API keys)
- Never exposed to server in plaintext (server receives RTMP URL which contains key as path param)
- Client constructs RTMP URL: `${ingestUrl}app/${streamKey}` — done in browser, then URL passed to server via WebSocket handshake or START message
- Server never logs the full RTMP URL (mask key in logs)

### Authentication to Streaming Server
- Optional shared secret token in WebSocket handshake
- If user runs server locally, no auth needed (localhost only)
- If remote server, recommend using `wss://` and token auth

### Rate Limiting & Abuse Prevention
- Server should limit max concurrent streams per token/IP
- Optional max duration (e.g., 8 hours) to prevent runaway processes

## 10. Open Questions / Risks

| Risk | Mitigation |
|------|------------|
| WebCodecs encoding too slow for real-time | Use hardware-accelerated encoder; test on target hardware; allow bitrate/fps adjustment |
| Audio-video sync drift | Use timestamp alignment; monitor PTS from encoder and Web Audio |
| Canvas capture capturing UI overlays (cursor, etc) | Option to hide cursor; capture only canvas content not browser chrome |
| Browser support: Safari lacks WebCodecs | Fallback to MediaRecorder? Lower quality; document Chrome/Edge requirement for streaming |
| Server scalability: One stream per FFmpeg process | Acceptable for single broadcaster; future: process pooling or WebRTC SFU |
| Twitch stream key leakage | Use secure storage; never send key to any server except Twitch via RTMP URL constructed client-side in server handshake |
| FFmpeg not installed on user system | Provide Docker image; document installation; check at server startup |
| Latency too high (5+ seconds) | Optimize encoder settings (low-latency preset); tune buffer sizes; measure and document expected latency (~3s) |

## 12. Alternatives Considered

### A. Use MediaRecorder + WebSocket
- **Pros:** Simpler API, no WebCodecs complexity
- **Cons:** No hardware acceleration; higher CPU; less control over bitrate/quality; real-time performance uncertain
- **Decision:** Not suitable for professional quality; discard

### B. Use WebRTC + WHIP to third-party service
- **Pros:** Standardized; built-in congestion control
- **Cons:** Requires WHIP-compatible server that converts to RTMP (not trivial); Twitch doesn't support WHIP natively
- **Decision:** Future enhancement; v1 uses WebSocket + FFmpeg

### C. Cloud-based streaming service (Mux, Vimeo Live)
- **Pros:** No server to manage; API-driven
- **Cons:** Cost; vendor lock-in; less control; requires account
- **Decision:** Keep architecture self-hosted for privacy/control; may add integrations later

### D. Use existing `ffmpeg.wasm` in browser to encode directly to RTMP?
- **Pros:** No server needed
- **Cons:** `ffmpeg.wasm` cannot do real-time streaming well; high memory; performance poor; RTMP not reliable over WASM
- **Decision:** Impractical; discard
